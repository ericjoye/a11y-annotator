How to Document Accessibility Issues for Client Audit Reports

July 13, 2026 · 7 min read

You've run the scans, found the issues, and now you need to communicate the findings to a client who may not know what WCAG stands for. How you document accessibility issues determines whether they actually get fixed — or sit in a spreadsheet until the next audit.

Based on feedback from dozens of freelance auditors and agency accessibility leads, here's the documentation workflow that gets results.

The Problem with Traditional Audit Reports

Most accessibility audit reports follow the same pattern: a 30-page PDF with screenshots, WCAG criterion references, and a severity rating. The client reads it once, forwards it to their developer, and the developer has to cross-reference each issue against the actual site to figure out what to fix.

This creates three problems:

  1. Context is lost — a screenshot shows where an issue is, but the developer still has to find the element in the codebase
  2. Issues get re-interpreted — the developer guesses what "add alt text to image" means for each of the 20 images flagged
  3. No clear completion signal — there's no obvious way for the client to confirm a fix was actually made

Better Approach: Annotated Issue Reports

The most effective audit documentation format makes each issue unambiguous. Here's what that looks like:

1. Pin Issues on the Page, Not in a Report

Instead of pasting screenshots into a document, use a tool that marks issues on the page itself. When each accessibility failure has a visual pin on the exact element, the developer sees the problem in its actual context — not cropped in a screenshot that may be out of date.

2. Include Both the Problem and the Fix

Each documented issue should state:

  • The element — CSS selector or a descriptive path (e.g., "the 'Submit' button in the contact form")
  • The violation — specific WCAG criterion and what failed (e.g., "1.4.3 Contrast ratio 3.2:1, needs ≥ 4.5:1")
  • The fix — actionable change (e.g., "Change button background to #2ea043")

3. Export to the Developer's Format

Different teams use different tools. A good audit documentation workflow supports multiple export formats:

  • Markdown — paste directly into GitHub/GitLab/Jira tickets
  • PDF — attach to a formal audit deliverable
  • CSV/JSON — import into project management tools for tracking

Template: Single Issue Documentation

Here's a reproducible format for documenting one accessibility issue:

Issue #: CONTRAST-003
Page: /pricing.html
Element: h2 "Enterprise Plan"
WCAG Criterion: 1.4.3 Contrast (Minimum) — AA
Current: #7a7a7a on #ffffff = 4.1:1
Required: ≥ 4.5:1 for normal text
Fix: Change color to #666666 or darker
Selector: .pricing-card h2

This format takes 30 seconds to write and saves the developer 5 minutes of hunting. Multiply by 30 issues and you've saved 2.5 hours per audit cycle.

Workflow: From Scan to Client Deliverable

  1. Scan the page with a local accessibility checker that pinpoints issues on the actual elements
  2. Review each pin — confirm the automated check is correct (false positives happen, especially on dynamic content)
  3. Add context notes — the automated pin gives you the what and where; you add the "how to fix"
  4. Export as Markdown — the annotated pins become structured issue entries
  5. Send to client with a cover note summarizing severity distribution: critical, major, minor

What Clients Actually Want to See

Most clients care about three numbers:

  1. Total issues found — scope of work
  2. Critical issues — things that block AA compliance (contrast failures, missing alt text on key images, keyboard traps)
  3. Estimated fix time — so they can budget the work

If you present your audit report with a clear severity breakdown and per-issue fix instructions, the client can estimate the work immediately — and you're more likely to get the remediation contract since you've already done half the documentation.
